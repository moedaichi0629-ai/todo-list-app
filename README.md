# ToDoリスト

ブラウザだけで使える、シンプルで見やすいToDoリストWebアプリです。

## アプリ概要

タスクの追加・完了・削除ができるシンプルなToDo管理アプリです。入力したタスクは`localStorage`に保存されるため、ブラウザを閉じたり再読み込みしたりしてもデータが消えません。

## アプリURL

https://todo-list-app-six-blue.vercel.app/

## 実装した機能

- タスクの追加(入力欄に文字を入れて「追加」ボタン、またはEnterキー)
- タスクの完了マーク(チェックボックスでON/OFF、完了時は取り消し線で表示)
- タスクの編集(「編集」ボタンで入力欄に現在の内容を表示し、「保存」で更新・「キャンセル」で中断)
- タスクの削除
- 期限日の設定(追加時・編集時にdate入力で指定、未入力なら期限なし)
- 期限日に応じた色分け表示(期限切れは赤、本日締切はオレンジ、それ以外はグレー)
- `localStorage`によるデータの永続化(ブラウザを閉じても消えない、編集内容・期限日も保存される)
- 空文字・空白のみのタスクは追加・保存できないようにするバリデーション
- 残りタスク数 / 全タスク数の表示

## 機能一覧

| 機能 | 操作方法 | 補足 |
|---|---|---|
| タスク追加 | 入力欄にテキストを入力して「追加」ボタン、またはEnterキー | 空文字・空白のみは追加不可 |
| 完了マーク | タスク左のチェックボックスをクリック | 完了時は取り消し線＋文字色を薄く表示 |
| タスク編集 | タスク右の「編集」ボタンをクリック | 上部の入力欄に現在の内容が入り、ボタンが「保存」に変化。編集中の行は青枠でハイライト表示 |
| 編集の保存 | 編集中に入力欄を書き換えて「保存」ボタン、またはEnterキー | 空文字・空白のみでは保存不可(編集状態を維持) |
| 編集のキャンセル | 編集中に「キャンセル」ボタン、またはEscキー | 入力内容を破棄して編集前の状態に戻る |
| タスク削除 | タスク右の「削除」ボタンをクリック | 削除前の確認なしで即削除。編集中のタスクを削除した場合は編集状態も自動的に解除 |
| 期限日の設定 | 上部フォームの📅日付欄で選択(追加時・編集時どちらも可) | 未入力のままでも追加・保存可能(期限なし) |
| 期限日の表示 | 特別な操作は不要 | 各タスクの下に📅アイコン付きで表示。期限切れは赤バッジ「期限切れ」、本日締切はオレンジバッジ「本日まで」、それ以外はグレー文字、期限なしは非表示 |
| データ永続化 | 特別な操作は不要 | `localStorage`に自動保存され、リロード・再訪問後も保持(編集結果・期限日も同様) |
| 残数表示 | 特別な操作は不要 | 「残り○件 / 全○件」を一覧下部に表示 |

## 実装手順

1. `create-next-app`でNext.js(JavaScript / App Router / Tailwind CSS構成)のプロジェクトを作成
2. `app/page.js`にToDoアプリ本体を実装
   - `useState`でタスク一覧・入力値を管理
   - `useEffect`で初回マウント時に`localStorage`からタスクを読み込み
   - タスクの状態(`tasks`)が変化するたびに`localStorage`へ書き込み
   - 追加・完了切り替え・削除の各処理を実装
3. Tailwind CSSでカード型レイアウト・配色・レスポンシブ対応を調整
4. `npm run dev`でローカル動作確認、`npm run lint` / `npm run build`で品質チェック
5. Playwrightを使ってブラウザ操作(追加・完了・削除・リロード後の永続化・モバイル表示)を自動テストして動作確認
6. `git init` → コミット → GitHubに新規リポジトリを作成してpush
7. VercelでGitHubリポジトリをImportし、デプロイして公開URLを取得
8. タスク編集機能を追加実装
   - `useState`で編集対象タスクのID(`editingId`)を管理し、`null`のときは「追加モード」、IDが入っているときは「編集モード」として扱う
   - 上部の入力フォームを追加・編集で共用し、編集モード開始時に対象タスクのテキストを入力欄へセット
   - 保存・キャンセル時に`editingId`をリセットし、タスク削除時にも編集中IDと一致すれば自動解除
   - 更新後のタスク一覧はそのまま既存の`localStorage`保存の仕組みに乗るため、保存処理の追加実装は不要
   - 再度`npm run lint` / `npm run build` / Playwrightでの動作確認を実施
9. 期限日設定機能を追加実装
   - タスクのデータ構造に`dueDate`(`"YYYY-MM-DD"`形式の文字列、または`null`)を追加
   - 上部フォームに`type="date"`の入力を追加し、追加・編集どちらの操作でも指定できるように共通化
   - 今日の日付と`dueDate`を文字列比較する`getDueDateStatus`関数を実装し、「期限切れ / 本日 / それ以降」を判定してバッジの色を出し分け
   - 既存の`localStorage`データ(`dueDate`を持たないタスク)でもエラーにならないよう、`dueDate`が無い場合は判定・表示処理をスキップして互換性を確保
   - Playwrightで期限切れ・本日・将来・期限なしの4パターンと、既存データとの互換性・編集による期限日変更・モバイル表示を確認

## 工夫した点

- 中央寄せのカード型レイアウトで、フォームとタスク一覧を1画面に収めてわかりやすくした
- 完了したタスクは文字色を薄くし取り消し線を付けることで、一覧を見ただけで進捗がひと目でわかるようにした
- 削除ボタンを目立たせすぎない赤系の配色にし、誤操作時にも視認しやすくした
- タスクが1件もないときは案内メッセージを表示し、初めて使うユーザーが迷わないようにした
- スマートフォンの画面幅でもボタンや文字が崩れないようレスポンシブ対応した
- 編集中の行を青枠・薄い背景色でハイライトし、一覧のどのタスクを編集しているか一目でわかるようにした
- 追加用フォームを編集時にも共用することで、画面上に入力欄を増やさずシンプルな構成のまま編集機能を追加した
- 編集モード中は「保存」ボタンの隣に「キャンセル」ボタンを表示し、Escキーでもキャンセルできるようにして誤操作から復帰しやすくした
- ボタンが増えても崩れないよう`flex-wrap`を使い、狭い画面ではボタン部分が自動的に折り返されるようにした
- 期限日バッジに📅アイコンを添えて視認性を高め、期限切れ(赤)・本日締切(オレンジ)は背景色付きのバッジにして、期限が近いタスクほど目立つようにした
- 期限日バッジの文字が狭い画面で縦に折り返されないよう`whitespace-nowrap`を指定し、スマホでも崩れず1行で表示されるようにした

## 学んだこと

- Reactの`useEffect`で外部システム(ブラウザの`localStorage`)と状態を同期させる際の注意点。初回読み込み前に空配列で上書き保存してしまわないよう、読み込み完了フラグを持たせる必要があること
- Next.jsのApp Routerでのクライアントコンポーネント(`"use client"`)の使いどころ
- Tailwind CSSでレスポンシブデザインを組む際、要素を`flex`で並べると子要素が意図せず縮んでしまう(`shrink-0`が必要になる)ケースがあること
- GitHub CLI(`gh`)とVercelを連携させることで、コードのpushからデプロイまでの一連の流れをスムーズに行えること
- 「追加」と「編集」のように似た操作は、別々のUIを作るより1つのフォームを状態(`editingId`)で出し分ける方が、コードの重複が減り既存の保存処理もそのまま再利用できること
- UIの状態遷移(追加中/編集中)が増えるほど、ボタンの表示・非表示や文言の切り替え漏れが起きやすいため、Playwrightで一通りのシナリオ(保存・キャンセル・空文字・削除との組み合わせ)を自動確認しておくと安心できること
- 日付は`"YYYY-MM-DD"`形式の文字列同士であれば、Dateオブジェクトに変換しなくても文字列比較(`<` / `===`)だけで前後関係を正しく判定できること
- 既存データとの互換性を保つには、新しいフィールド(`dueDate`)が存在しないケースを「値が無い」として自然に扱えるように書く(存在チェックを厳密にしすぎない)ことで、特別なマイグレーション処理を書かずに済むこと
- ブラウザのテスト自動化(Playwright)でタイムゾーンを扱う際は、`toISOString()`はUTC基準になるためアプリ側のロジック(ローカル日付)とズレることがあり、日付の生成方法をアプリ側と揃える必要があること
